## Faza 1: Ship narrow (~4-5h)

### A. `DerivativeFilingDetector` (pure function)

`src/sec-filings/utils/derivative-filing-detector.ts` — pure function bez side-effects, testowalna w izolacji (analogicznie do `extract-guidance-status.ts` / `missing-data-detector.ts`).

```ts
export interface DerivativeDetection {
  isDerivative: boolean;
  matchedPhrases: string[];      // dla audit log
  priorReferenceDate?: string;   // jeśli regex wyłuska datę "dated March 31, 2026"
}

export function detectDerivativeFiling(
  filingText: string,
  catalystType: string,
): DerivativeDetection;
```

**Phrases (PL+EN, case-insensitive)**:

```
- /previously announced/i
- /in connection with the (?:previously announced|merger|acquisition)/i
- /pursuant to the (?:merger|acquisition|purchase) agreement dated/i
- /pursuant to the agreement and plan of merger dated/i
- /the merger agreement (?:was |dated )/i
- /as previously disclosed/i
- /in furtherance of the previously announced/i
```

**Trigger logic**:
- `isDerivative = matches.length >= 1 AND catalystType IN {contract, merger, financing, other}`
- 2.02 (earnings) → zawsze `isDerivative=false` (earnings nie są derivative w tym sensie)
- `priorReferenceDate` opcjonalnie z regex `dated (\w+ \d+,? \d{4})`

### B. Prior filing DB lookup (`Form8kPipeline.onFiling`)

Po `detectItems()`, przed `extractItemText()`, dla itemów non-2.02:

```ts
const derivative = detectDerivativeFiling(filingText, mainItem);
let priorContextBlock: string | null = null;
let priorFiling: SecFiling | null = null;

if (derivative.isDerivative) {
  const priorRelated = await this.filingRepo.find({
    where: {
      symbol: payload.symbol,
      formType: '8-K',
      filingDate: MoreThan(new Date(Date.now() - 90 * 24 * 3600 * 1000)),
      gptAnalysis: Not(IsNull()),
    },
    order: { filingDate: 'DESC' },
    take: 5,
  });

  // filter: |conviction| > 0.5 AND catalyst_type matching
  priorFiling = priorRelated.find(f =>
    Math.abs(f.gptAnalysis.conviction) > 0.5
    && ['merger', 'contract', 'financing'].includes(f.gptAnalysis.catalyst_type)
  ) ?? null;

  priorContextBlock = formatPriorContext(derivative, priorFiling);
}
```

### C. `priorContextBlock` format (inject do prompta)

```
## PRIOR CONTEXT (extracted from DB — TRUST OVER OWN INFERENCE):

This 8-K filing references a previously disclosed event. Market has already had
opportunity to price the original announcement.

Derivative markers detected:
  • "<matched phrase 1>"
  • "<matched phrase 2>"

{if priorFiling}
Prior related filing (from DB):
  - Date: YYYY-MM-DD (N trading days ago)
  - Item: X.XX, catalyst: <type>
  - Prior conviction: ±X.XX, direction: <dir>
  - Prior summary: "<prior.gptAnalysis.summary>"

CONSTRAINT: Your conviction MUST reflect ONLY incremental information vs the
prior event. The current filing operationalizes (financing, regulatory step,
closing) something the market already knows. Default conviction range: ±0.1
to ±0.4. Use higher magnitude ONLY if current filing reveals materially worse
or better terms than prior disclosure.
{else}
No prior filing in DB matches — but derivative markers indicate this filing
references an external prior event (e.g. news, target company filing).
CONSTRAINT: Apply conservative conviction (±0.1 to ±0.5) since market likely
has priced the original event.
{endif}
```

### E. Post-LLM derivative guard

`src/sec-filings/utils/derivative-conviction-guard.ts` (pure function, exported):

```ts
export interface DerivativeGuardDecision {
  cap: number | null;            // null = no cap
  reason: string;                // 'derivative_same_dir' | 'derivative_opposite' | 'derivative_no_prior'
  details: string;               // log-friendly
}

export function shouldCapForDerivative(
  isDerivative: boolean,
  priorConviction: number | null,
  currentConviction: number,
): DerivativeGuardDecision;
```

**Reguły** (priority top-down):

| Reguła | Warunki | Cap |
|---|---|---|
| R1 same-direction | `isDerivative && priorConviction != null && sign(curr)==sign(prior) && \|curr\|>0.5` | **0.3** |
| R2 opposite-direction | `isDerivative && priorConviction != null && sign(curr)*sign(prior)<0 && \|curr\|>0.7` | **0.5** |
| R3 no-prior | `isDerivative && priorConviction == null && \|curr\|>0.6` | **0.5** |
| R4 default | inaczej | no cap |

**BIIB case (R1)**: prior BIIB/Apellis 31.03 → conviction ~-1.0 negative (debt + dilution narrative); current 14.05 credit agreement → conviction -1.10 negative → same direction, |curr|=1.10>0.5 → **cap 0.3**.

## Ryzyka / blefy (samokrytyka, Plan v3)

1. **Regex false positives**: "previously announced today" (nowa wiadomość ALE phrase match)
   - Mitygacja: testowany edge case, prompt instruction "use higher magnitude IF current reveals materially worse/better terms"
   - Faza 2: monitor logi → revert jeśli >1 false positive na 14d

2. **Prior filing miss — counterparty case**: BIIB/Apellis deal został ogłoszony przez **Apellis** (target), Form 8-K acquirera (Biogen) z 31.03 może mieć inny catalyst_type
   - Mitygacja: R3 (no_prior) reguła cap 0.5 — phrase match wystarczy nawet bez prior w DB
   - Faza 2: monitor ratio R1:R2:R3 — jeśli R3 dominuje, to znak że prior lookup ma niski recall

3. **N=1 sample dla cap thresholds**: BIIB to jedyny case w handover (22.05.2026)
   - Mitygacja: Faza 1 ship → Faza 2 obserwacja zbierze więcej (M&A activity w healthcare ~5-10 deals/Q)
   - Plan v3 framework: NIE deployujemy z N=1 jako finalne — Faza 1 to deployment + obserwacja

4. **Edge case real incremental info**: "pursuant to merger agreement... AND additional $500M unexpected debt covenant"
   - R2 (opposite direction) próbuje to złapać konserwatywnie
   - Faza 2: jeśli ≥2 cap'y na real incremental info → rekalibracja R2 threshold |conv|>0.8 zamiast 0.7

5. **catalyst_type mismatch**: prior=merger, current=contract (Item 1.01) — czy lookup powinien filtrować po catalyst?
   - Decision: filter na `['merger', 'contract', 'financing']` set — wszystkie 3 są semantycznie powiązane w follow-up scenario
   - Faza 2: monitor czy prior matching dobrze trafia

6. **90-day window arbitrary**: dlaczego 90 a nie 60 lub 180?
   - Rationale: typowy M&A deal close cycle 30-90 dni od announcement (regulatory + financing). 90 łapie financing follow-ups + closing 8-K + ewentualne CVR triggers
   - Faza 2: monitor priorFiling.daysAgo distribution — jeśli >70% trafia w 0-60d → szutaj 60d, brak benefit z 90
