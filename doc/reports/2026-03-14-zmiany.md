## 1. Wyłączenie StockTwits z eskalacji GPT

**Plik**: `src/sentiment/sentiment-processor.service.ts` (linie 133-140, 188-192)

**Problem**: 78.1% conviction scores z GPT było praktycznie zerowe (-0.01 do 0.01). Przyczyna: GPT przypisuje StockTwits `source_authority = 0.15`, a conviction jest mnożone przez source_authority — co zeruje wynik. W praktyce 3 133 z 3 773 wywołań GPT w tygodniu (83%) generowało wartość ~0. Koszt Azure VM bez żadnego zwrotu w jakości sygnału.

**Zmiana**:
```typescript
// PRZED:
const shouldEscalate =
  tier === 1 || (tier === 2 && this.azureOpenai.isEnabled());

// PO:
const isGptEligibleSource = source !== DataSource.STOCKTWITS;

const shouldEscalate =
  isGptEligibleSource &&
  (tier === 1 || (tier === 2 && this.azureOpenai.isEnabled()));
```

Dodano też nową gałąź logowania — StockTwits Tier 1/2 raportuje status `FINBERT_ONLY` z informacją:
```typescript
} else if (!isGptEligibleSource && tier <= 2) {
  pLog.status = 'FINBERT_ONLY';
  this.logger.debug(
    `Skip AI (${source} nie kwalifikuje się do GPT): ${symbol} — ${reason}`,
  );
}
```

**Efekt**:
- Redukcja wywołań Azure VM o ~83% (z ~3 773 do ~640 tygodniowo)
- Brak straty jakości sygnału — StockTwits i tak dostawał conviction ~0 z GPT
- StockTwits nadal przechodzi przez FinBERT (1. etap) — effectiveScore = FinBERT score
- FINNHUB i SEC EDGAR nadal eskalowane do GPT normalnie (Tier 1 + Tier 2)

**Dane uzasadniające** (7-13 marca):

## 2. Wyczyszczenie 500 failed jobów BullMQ

**Komenda**: `redis-cli ZREMRANGEBYSCORE "bull:sentiment-analysis:failed" "-inf" "+inf"`

**Problem**: 500 jobów w zbiorze `failed` kolejki `sentiment-analysis`. Wszystkie miały ten sam powód: `"fetch failed"` z timestampem 20 lutego 2026 (~20:10 UTC). Jednodniowy incident — prawdopodobnie Azure VM niedostępna lub problem sieciowy. BullMQ nie ma TTL na zbiór failed, więc joby leżały 3 tygodnie.

**Efekt**: Zbiór `failed` wyczyszczony do 0. Brak wpływu na bieżące przetwarzanie — joby były z 20.02, dawno nieaktualne.
